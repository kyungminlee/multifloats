# multifloats vs Boost.Multiprecision — operation matrix

Generated by `scripts/build_op_matrix.py`. One row per operation that
either side implements; a `—` cell means that side does not implement
the op or the harness does not test/time it. `mf`/`boost` time columns
are wall-clock seconds on the same workload (lower is better).

Source data: `cpp_fuzz 1000000 42`, `boost_dd_fuzz 1000000 42`,
`cpp_bench`, `boost_dd_bench` (all matched seed). Re-generate with:

```sh
cmake -DBUILD_TESTING=ON -DMULTIFLOATS_BUILD_BENCH=ON \
      -DMULTIFLOATS_BUILD_BOOST_COMPARE=ON -S . -B build
cmake --build build -j --target cpp_fuzz cpp_bench boost_dd_fuzz boost_dd_bench
./build/cpp_fuzz       1000000 42 > /tmp/mf_fuzz.txt
./build/boost_dd_fuzz  1000000 42 > /tmp/bd_fuzz.txt
./build/cpp_bench                  > /tmp/mf_bench.txt
./build/boost_dd_bench             > /tmp/bd_bench.txt
python3 scripts/build_op_matrix.py > doc/developer/BOOST_OP_MATRIX.md
```

Boost's `cpp_double_double` backend natively implements only
`+ − × ÷ sqrt fabs exp log pow floor ceil frexp ldexp ilogb`;
everything else routes through `boost::math::*` generics on top of
the backend, which explains the boost-side precision/speed gaps in
the trig / hyperbolic / Bessel sections.

Naming notes:
- multifloats labels its FMA fuzz row `fma_cxx`/`fmadd`; boost uses
  `fma`. Reported below as a single `fma` row.
- multifloats fuzzes `gamma`/`lngamma` labels for what boost calls
  `tgamma`/`lgamma`. Merged for readability.
- multifloats `sincos`/`sinhcosh` produce two stat rows (`_s` sin
  component and `_c` cos component); the bench measures the joint
  call. Both shown.
